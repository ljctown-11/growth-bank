## 0. 落地约束（给架构师/工程师的前置条件）

- **技术栈不变**：原生 HTML+CSS+JS（ES Module 多文件），无框架；PWA（manifest + sw.js）；localStorage（结构化 STATE）+ IndexedDB（媒体 blob，复用 `features/media.js`）。

- **现有 UI 闸门保持**：`toggleTask` 中"未设宝贝信息则拦截"逻辑不动；新增/改造入口在未设宝贝时一律禁用/隐藏。

- **测试底座**：Vitest 单测（QA 严过关负责）；3 个真 bug 需补充浏览器/单元验证（见 §7）。
- **交付**：不推送 GitHub/Gitee，仅走 CloudStudio 部署测试验证。

## 1. 产品目标（本轮）

**一句话目标**：把"找得到、看得清、不卡顿"做到位——让任务灵感更显眼且可用、让成长档案一页看完成就与作品、让每周复盘以"周"为单位沉淀，并消除 3 个让操作卡住的真实 bug。

**本轮 3 条成功指标（可观测）**：
1. **任务灵感可用率 100%**：点击"设为今天任务"→ 当日任务列表出现该条且可勾选得分；灵感弹层任意次打开/关闭均不残留遮罩。（对应修 C2、C3）
2. **档案一页可达**：成长树、徽章墙、周表、作品在档案页单页纵向可达，桌面端"我的成就"两列并排；日历 Tab 连续打卡块移除后顶栏 pill 仍正确显示。
3. **设置闭环无卡死**：设置宝贝信息点"确认"后无论密码成功/取消，弹层都能正确收掉或回到设置，无残留遮罩。（对应修 E）

## 3. 用户故事（按模块）

### A 日历 Tab 瘦身
- **US-A1**：作为孩子/家长，我在日历 Tab 只想看日历、当日成长地图和积分趋势，不想再下滑看重复的"连续打卡"（顶栏已有 🔥 pill），也不想在这里看成长树/徽章（应进档案）。
  - 验收：日历 Tab 仅含 暑假日历 + 成长地图 + 积分趋势图 3 块；顶栏 🔥 连续 N 天 pill 正常显示且与之前一致；原"连续打卡/成长树/徽章墙"不再出现在日历 Tab。

### B 任务灵感（改名 + 显眼 + 修 bug）
- **US-B1**：作为孩子，当不知道做什么时，能在打卡页显眼位置看到"小芽帮你挑一个"的引导，而不是一个不显眼的角落按钮。
  - 验收：打卡页心情行下方出现醒目引导卡/横条；文案含吉祥物语气（如"不知道做什么？小芽帮你挑一个🌱"）。
- **US-B2（修 C2）**：作为孩子，点"设为今天任务"后，该灵感立刻出现在当日任务列表且可勾选得分。
  - 验收：点击后任务列表新增该条（带 🌟灵感 标签），勾选后计入成长分；不再无反应。
- **US-B3（修 C3）**：作为孩子，打开/关闭灵感弹层多次均正常，不会出现"闪一下关不掉、再点才消失"。
  - 验收：连续打开关闭灵感弹层不残留遮罩；用统一 `openModal`（F）保证单例。
- **US-B4**：作为孩子，补卡（选中过去日期）时不该被引导加新任务。
  - 验收：选中非"今天"日期（出现补卡横幅）时，任务灵感入口隐藏或置灰；仅选中"今天"时显示。

### C 成长档案页重组（方案 A）
- **US-C1**：作为家长，我希望在档案页一页看完孩子的成就（成长树+徽章）、每周复盘、成长作品，不再在子 Tab 间切换。
  - 验收：档案页单页纵向含 4 模块；原"每周复盘/成长作品"子 Tab 删除；桌面端"我的成就"两列并排，手机端单列堆叠。
- **US-C2**：作为家长，成长作品以"日期最前"呈现，且双排网格更紧凑。
  - 验收：每个作品卡日期作为醒目小标题在最前，下面才是任务名；桌面 2 列、手机 1 列。

### D 每周复盘改周表
- **US-D1**：作为家长，复盘以"第 N 周 + 日期区间"为单位查看，点某周展开 5 个填写框 + 本周心情，而不是按日期散落。
  - 验收：周表列出"第1周 7/1–7/7"等；点某周展开 5 框（最棒的事/遇到的困难/下周想调整/家长看见的进步/孩子希望的支持）+ 本周心情（😊😐😢）；本周心情独立存储，不与每日心情混淆。

### E 设置弹层（修 bug）
- **US-E1（修 E）**：作为家长，在设置宝贝信息点"确认"后，无论密码校验成功还是取消，界面都能干净收掉或回到设置，不会卡在"看不见但挡着"的弹层后。
  - 验收：点确认→若有家长密码则弹密码键盘且设置弹层先隐藏；密码成功→整体收掉；密码取消→返回设置弹层可继续编辑；不残留任何遮罩。

### F 统一弹层工具
- **US-F1**：作为开发，所有弹层用同一套"单例+统一关闭"机制，避免再出现叠加遮罩类 bug。
  - 验收：`openModal(id, contentBuilder)` 保证同 id 同时仅一个；统一支持 关闭按钮 / 遮罩点击 / Esc 关闭；灵感弹层、复盘详情、作品详情改造后复用。

## 6. 待确认问题（需升级给用户 / 架构师）

### Q1（升级架构师，P1/D4）旧复盘数据迁移策略
- 现状：`STATE.reviews` 仍是"按日"数组，每条含 `date`/`weekKey`/`best`/`hard`/`next`/`parent`/`support`；`core/data.js` `loadData` **已**做了"同 `weekKey` 合并为最新内容"的预处理（L13-36）。
- 待定：新结构改为"按周记录表（weekKey→weekIndex + dateRange + 5 字段 + mood）"。需架构师确认：
  1. 是否直接复用现有 `weekKey` 作为周记录主键，并补 `weekIndex`/`dateRange`/`mood`？
  2. 旧"按日多条"是否丢弃、仅保留合并后的周快照？还是保留每日明细另存？
  3. `dateRange` 如何由 `weekKey`（周一）还原为 `"7/1–7/7"` 区间文案？
- 建议：保持"同周合并"思路，新增字段补全；旧数据不丢、不报错。

### Q2（升级用户 + 架构师，D1/D2）周表日期区间与"第 N 周"口径
- 待定：① 暑假起止日期（用于"第1周"锚点）是固定 7/1 起，还是取日历可打卡首日？② 周划分是否沿用 `getWeekKey`（周一为一周起点）？③ "第 N 周"编号是否从暑假第 1 个周一开始顺排？
- 影响：周表每一行的标题与区间文案、以及 `weekIndex` 计算。
- 建议：以"暑假首日所在周为第1周"，沿用周一为起点；需用户确认暑假起止。

### Q3（升级架构师，B3/C2）灵感任务 id 方案（最小改动 vs 全链路统一）
- 现状：`renderIdeaLibrary` 按钮 `data-idea-id` 用 `ideaIdOf(idea)`（哈希）；`addIdeaAsTask` 也用 `ideaIdOf` 作为 `day.tasks` 键；"已添加"判定同样用哈希。三者一致但**都和静态 id 对不上**。
- 修复方向（团队已定）：按钮查找改静态 id 即可命中。但需架构师确认二选一：
  - **方案甲（最小改动）**：仅按钮 `data-idea-id` 改静态 id，`findIdeaById` 命中；`addIdeaAsTask` 与"已添加"判定仍用哈希（内部一致）。改动最小、风险低。
  - **方案乙（全链路统一）**：`day.tasks` 键也改静态 id，需处理已存哈希键的旧灵感任务数据迁移。
- 建议：采用方案甲（本次只修 bug，不引数据迁移风险）；若后续统一再单独排期。

### Q4（升级架构师，E1/E2）设置弹层"先隐藏"的具体交互
